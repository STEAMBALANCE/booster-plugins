# Orders modal ‚Äî Steam key activation protocol

The "–ú–æ–∏ –∑–∞–∫–∞–∑—ã" (My Orders) window is a `steambalance.cc/booster/orders`
page embedded as an iframe inside the SteamBooster wrapper window. It talks
to the `booster-checkout` plugin over the framework embed bus
(`window.postMessage`). This document is the contract the orders page
implements to activate Steam product keys via the framework
(`sb.keys.activate`).

It mirrors the native `window.SteamBooster.activateKey()` bridge used by the
checkout payment page, but over the embed bus (the native bridge is not
available inside the wrapped iframe).

## Envelope

All embed-bus messages carry the standard envelope:

    { "__sbEmbed": true, "v": 1, "type": "sb:event", "name": "...", "data": {...} }

The page must already be wired into the embed handshake (`sb:ready` ‚Üí
`sb:embed` ‚Üí `sb:embed-payload`); activation messages reuse the same channel.

## Request (page ‚Üí plugin)

    {
      "__sbEmbed": true, "v": 1, "type": "sb:event",
      "name": "activate-key",
      "data": { "requestId": 1, "key": "XXXXX-XXXXX-XXXXX" }
    }

- `requestId` ‚Äî finite number, generated by the page, used to correlate the
  reply. Reuse must be avoided while a request is in flight.
- `key` ‚Äî non-empty string, 1..256 chars.

Post it to the parent window (the wrapper relays it to the plugin):

    window.parent.postMessage(request, 'https://steamloopback.host');

## Response (plugin ‚Üí page)

Exactly one of `outcome` / `error` is present:

    // pipeline ran ‚Äî business success OR business failure inside outcome.ok
    { ..., "name": "activate-key-result", "data": { "requestId": 1, "outcome": { /* ActivateOutcome */ } } }

    // validation / transport / oversize failure
    { ..., "name": "activate-key-result", "data": { "requestId": 1, "error": "..." } }

Mapping (identical to the native bridge):

- `outcome` present ‚Üí resolve with the `ActivateOutcome`.
  `ActivateOutcome` is `{ ok: true, products, transactionId }` on success, or
  `{ ok: false, code, resultDetail, message }` on a business failure (key
  already owned, region-locked, invalid, etc.). A business failure is a
  resolved outcome, NOT an error. The canonical type lives in
  `@steambalance/booster-framework` (`api-types.ts` ‚Üí `ActivateOutcome` /
  `ActivateErrorCode`); treat that as authoritative over this summary.
- `error` present ‚Üí reject with `new Error(data.error)`. The two validation
  errors are the fixed literals below; transport/framework failures carry an
  implementation-defined message ‚Äî surface it, do not pattern-match on it.

## Required: page-side timeout

Reply delivery is best-effort. If the window is closed mid-activation, the
page receives nothing (the activation may still have consumed the key). The
page MUST arm its own timeout (recommended ‚â• 40 s, matching the framework
relay timeout) and treat expiry as "status unknown ‚Äî do not retry" (the same
contract as the native bridge's transport error). A consumed key must never
be silently re-submitted. (An over-cap reply is NOT a silent loss ‚Äî it is
downgraded to a delivered `error`; see Limits.)

## Limits

- Key: 1..256 chars (longer / empty ‚Üí `error: "invalid product key"`,
  activation is not attempted).
- `requestId`: must be a finite number. If it is absent, non-numeric, or
  non-finite, the plugin ignores the message and sends NO reply ‚Äî the page's
  own timeout fires. (A valid `requestId` with an invalid `key` still gets an
  `error` reply; only a bad `requestId` is dropped silently, since there is
  nothing to correlate a reply to.)
- Message size: 16 KB per message, both directions; oversize is dropped by
  the bus. Activation outcomes are tiny in practice; if a success reply would
  exceed the cap the plugin downgrades it to `error: "activation response
  too large"` so the page rejects deterministically instead of hanging.

## Minimal page-side client

    const pending = new Map();
    let nextId = 1;

    window.addEventListener('message', (e) => {
      if (e.origin !== 'https://steamloopback.host') return;
      const m = e.data;
      if (!m || m.__sbEmbed !== true || m.type !== 'sb:event' || m.name !== 'activate-key-result') return;
      const p = pending.get(m.data && m.data.requestId);
      if (!p) return;
      pending.delete(m.data.requestId);
      clearTimeout(p.timer);
      if (m.data.outcome) p.resolve(m.data.outcome);
      else p.reject(new Error((m.data && m.data.error) || 'activation failed'));
    });

    function activateKey(key) {
      return new Promise((resolve, reject) => {
        const requestId = nextId++;
        const timer = setTimeout(() => {
          pending.delete(requestId);
          reject(new Error('activation timeout ‚Äî status unknown, do not retry'));
        }, 40000);
        pending.set(requestId, { resolve, reject, timer });
        window.parent.postMessage(
          { __sbEmbed: true, v: 1, type: 'sb:event', name: 'activate-key', data: { requestId, key } },
          'https://steamloopback.host',
        );
      });
    }
